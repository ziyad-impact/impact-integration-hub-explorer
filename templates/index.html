01 — Who's Involved · The Three Parties

Three distinct audiences, three separate portals, one shared platform. Every interaction on impact.com traces back to one of these parties — and each has a different data scope, API surface, and set of goals.

Brand Portal
Brand Integrations Portal
For advertisers running affiliate & partner programs. Connects the impact.com platform to websites, eCommerce stores, and third-party tools.
impact-brand v8 → v13 Most content —
↓ Click to explore architecture
Architectural Role
Brand is the demand side of the marketplace. Brands own the conversion event and define commission rules via Contracts. Every integration decision here ultimately answers one question: did this partner deserve credit, and how much? The Brand API is the only portal that can submit Conversions — Partners and Agencies can only read the downstream results.
Economic Role
Brands are impact.com's paying customers. They pay a platform subscription fee to impact.com, and separately pay partner commissions — which impact.com tracks, calculates, and clears. The economic appeal: performance-based spend — brands only pay commissions when a tracked, attributed conversion actually occurs. No conversions, no commission cost.
How it relates
→ Partner Portal: Partners consume what Brand produces. Brand submits Conversions; Partner reads Actions. Brand creates Contracts; Partner reads commission outcomes. Brand publishes Deals and Promo Codes; Partner distributes them.
→ Agency Portal: Agency is a passthrough over Brand. An agency credential can call any Brand endpoint by injecting the managed brand's AccountSID into the URL path. No separate Brand API surface — just a credential model change.

Partner Portal
Partner Integrations Portal
For publishers & affiliates who earn commissions. Partners read account data, retrieve tracking links, sync catalogs, and export performance reports.
impact-publisher v11 → v15 API-heavy —
↓ Click to explore architecture
Architectural Role
Partner is the supply side. Partners are readers, not writers — the Partner API is almost entirely GET-based. Partners consume brand-side data (Actions, Contracts, Deals) and produce outbound links (Tracking Links). The one notable write is IdentityVerification for KYC onboarding. The asymmetry with Brand is intentional: the brand controls attribution, the partner consumes the output.
Economic Role
Partners are free to use the platform — impact.com does not charge publishers. The partner's income is entirely commission-based: they earn a share of sales, leads, or installs they drive to brand programs. The economic incentive is clear: more quality traffic → more conversions → more earnings. impact.com provides the measurement and payment infrastructure that makes partner income predictable and transparent.
How it relates
← Brand Portal: All data in the Partner API is downstream of Brand. Actions come from Brand Conversions. Contracts come from Brand commission rules. Catalogs are published by Brands. Partners have a read-only window into Brand's operational data.
Agency Portal: Agencies don't interact with Partners directly via API. The Agency passthrough model operates purely on the Brand side — Agencies manage Brands, not Partners.

Solution Provider Portal
Solution Provider / Agency Portal
For agencies managing multiple brand accounts. A Brand API passthrough allows agencies to access all brand endpoints on behalf of managed advertisers.
impact-agency v8.0 Passthrough model —
↓ Click to explore architecture
Architectural Role
Agency is a meta-layer over Brand, not a separate API surface. The key architectural insight: the Agency portal has almost no unique endpoints. It exists to solve a credential management problem — how can one set of API keys access N brand accounts? The answer is the passthrough model: /Agencies/{AgencySID}/Advertisers/{BrandID}/{BrandEndpoint}. One credential, any brand in your portfolio.
Economic Role
Agencies occupy a service layer between impact.com and brands. The agency pays platform fees to impact.com for their managed brand accounts. Their economic model is a management service fee charged to each brand client — they earn by providing program strategy, optimisation, and operations. The passthrough API model means one technical integration serves the entire client portfolio, keeping the agency's operational costs low as they scale.
How it relates
→ Brand Portal: Agency documentation explicitly defers to Brand docs for endpoint details — there's no duplication. The agency base URL is the only difference. This is a documentation strategy, not an API design choice.
Agency-only endpoints: The two truly unique Agency endpoints are Advertisers (list all managed brands) and Reports/ReportExport (aggregate reporting across the portfolio). Everything else is a passthrough.

02 — How Money Flows · The Economic Model

Brands pay a platform subscription to impact.com, then pay commissions directly to partners per conversion. impact.com earns from subscriptions — not by taking a cut of partner commissions. This separation is what makes the attribution trustworthy to both sides.

Why this structure works

This is the architectural answer to the trust problem in performance marketing. Brands need attribution they can verify. Partners need payments they can rely on. impact.com sits neutrally between them — earning from subscriptions, never from commissions — so its incentives align with growing the marketplace, not extracting from individual transactions.

06 — Doc Types · Guides vs. API Reference

Every portal ships two kinds of documentation: Integration Guides for task-oriented walkthroughs, and API Reference for precise endpoint specs. Knowing which to reach for saves significant time — guides answer "how do I do X?", reference answers "what does this parameter do?"

📖
Integration Guides (Guides Tab)
Task-oriented, narrative documentation that walks a reader through an integration end-to-end. Written procedurally: "do X, then Y, then Z." The goal is to complete a setup.
Voice Procedural, second-person, action verbs
Audience Developers & technical marketers doing a one-time setup
URL pattern /impact-brand/docs/<slug>
Reads Cover-to-cover, once
⚡

Endpoint-by-endpoint technical specification. Every endpoint's purpose, HTTP method, path, authentication, parameters, and example responses. Readers dip in and out — they don't read cover-to-cover.
Voice Precise, declarative, structured

URL pattern /impact-brand/reference/<endpoint-slug>

Plugin Integrations
~18 pages ▶
Architect's note
Plugins are pre-built connectors that handle UTT installation and Conversion submission automatically within a third-party platform. They remove the need to write custom integration code. Architecturally, they sit between the brand's eCommerce platform and impact.com's server-side Conversion API.
  • MMP (Mobile Measurement) — Adjust, AppsFlyer, Branch
  • CDP — Segment
  • CRM — HubSpot, Salesforce
  • e-Commerce — Shopify, BigCommerce, WooCommerce, Magento, Amazon, and more
  • Subscriptions & Payments — Stripe, Chargebee, Recurly
Tracking Integrations
~14 pages ▶
Architect's note
Three parallel paths to the same outcome — a Conversion in impact.com: UTT (JavaScript, client-side), Server-side API, and FTP batch upload. The first is easiest but cookie-dependent; the second is reliable but requires backend work; the third is for legacy/offline scenarios. GTM is a wrapper around UTT, not a separate mechanism.
  • Consent Mode — GDPR/cookie consent integration
  • API Tracking (Online Sale) — server-side implementation
  • FTP Tracking — batch file-based conversion upload
  • Google Tag Manager (GTM) — wrapper around UTT
  • JavaScript Tag (UTT) — client-side tracking
  • Card Linked Offers (Beta) — card-based tracking
  • Email Hashing & Security — customer data privacy

API Reference — Partner Endpoints
~12 resource groups ▶
Architect's note
Notice the overlap with Brand API: Accounts, Actions, Catalogs, Contracts, Jobs, ReportExport all exist in both portals. They're the same data, scoped to the authenticated party. A Partner reading Actions sees their credited conversions; a Brand reading Actions sees all conversions in their program.
  • Accounts — company metadata (name, website, contacts)
  • Actions — credited conversions eligible for payout
  • Tracking Links — programmatically generate deep links, ad links, vanity links

  • Contracts — view commission terms including Scheduled Terms
  • IdentityVerification — create/resume a KYC verification session
  • WithdrawalSettings — banking fields for payout configuration
  • Scoped Tokens — fine-grained API credential permissions

Brand API Passthrough
Key pattern ▶
Architect's note
The passthrough pattern solves a real problem: agencies manage 10–100+ brand accounts and can't maintain separate API credentials per client. The URL structure /Agencies/{SID}/Advertisers/{BrandID}/Actions is identical to the Brand URL except for the prefix. impact.com's API gateway swaps the auth context at the prefix — elegant and zero-duplication.
  • URL pattern — /Agencies/<SID>/Advertisers/<BrandID>/<BrandEndpoint>
  • All brand endpoints supported — full Brand API accessible via passthrough
  • Credential model — one Agency SID/AuthToken; target brand by AccountID param
  • Doc strategy — Agency docs point back to Brand API docs; no duplication

# impact.com Developer Documentation
 
> impact.com is a partnership automation platform connecting Brands, Partners, and Agencies
> through attribution tracking, commission management, and payment clearing.
 
## Brand Portal
 
- [Integration Guides](https://developer.impact.com/brand/docs): Setup for advertisers — UTT installation, Contracts, tracking, reporting
- [API Reference v13](https://developer.impact.com/brand/api): REST API — Conversions, Actions, Contracts, Catalogs, Deals, MediaPartners
 
## Partner Portal
 
- [Integration Guides](https://developer.impact.com/partner/docs): For publishers — tracking links, reporting, payment setup
- [API Reference v15](https://developer.impact.com/partner/api): REST API — Actions, Deals, Catalogs, ReportExport, TrackingLinks
 
## Agency Portal
 
- [Integration Guides](https://developer.impact.com/agency/docs): For agencies managing multiple brand accounts
- [API Reference v3](https://developer.impact.com/agency/api): REST API — passthrough over Brand API with Advertisers endpoint
 
## Authentication
 
HTTP Basic Auth — AccountSID (username) + AuthToken (password).
Agency callers inject managed brand AccountSID into the URL path.
 
## Key Terms
 
- UTT: Universal Tracking Tag — JS snippet installed in <head> on all brand site pages
- irclickid: Click ID appended to partner referral URLs for attribution tracking
- Contract: Commission rules defining payout structure between a brand and a specific partner
- Action: Tracked conversion event (sale, lead, install) attributed to a partner, eligible for payout
